# Frontend 구현 계획서

## 1. 프론트엔드 설계 철학

### 1.1 사용자 중심 원칙
- **신뢰성**: 같은 입력은 항상 같은 결과 (PRD 핵심 목표 반영)
- **명확성**: 입력 과정이 직관적이고 혼란스럽지 않음
- **사고 촉발**: 결과가 단순 정보가 아닌 깊은 사고를 유도하는 디자인
- **접근성**: 모든 사용자가 쉽게 사용할 수 있는 인터페이스
- **성능**: 빠른 응답과 부드러운 인터랙션

### 1.2 디자인 원칙
- **미니멀리즘**: 불필요한 요소 제거, 핵심에 집중
- **타이포그래피 중심**: 텍스트가 주인공, 읽기 쉬운 구조
- **감정적 중립**: 감정적 위로나 동기부여 배제, 객관적 진단에 집중
- **시각적 계층**: 정보의 중요도에 따른 명확한 시각적 구분
- **일관성**: 전체 페이지에서 통일된 디자인 언어

## 2. 기술 스택

### 2.1 필수 기술
- **프레임워크**: Next.js 14+ (App Router)
- **언어**: TypeScript
- **스타일링**: Tailwind CSS
- **폼 관리**: React Hook Form
- **유효성 검증**: Zod
- **상태 관리**: React Context API (또는 Zustand)

### 2.2 추가 라이브러리
- **아이콘**: Lucide React (또는 Heroicons)
- **애니메이션**: Framer Motion (선택)
- **토스트 알림**: Sonner (또는 react-hot-toast)

## 3. 페이지별 상세 설계

### 3.1 메인 페이지 (`/`)

#### 3.1.1 목적
- 서비스 소개 및 진단 시작 유도
- 사용자에게 간단하고 명확한 첫인상 제공

#### 3.1.2 레이아웃 구조
```
┌─────────────────────────────────────┐
│         Header (로고/제목)           │
├─────────────────────────────────────┤
│                                     │
│      서비스 소개 섹션                │
│      (간단한 설명, 핵심 가치)         │
│                                     │
│      ┌─────────────────────┐        │
│      │   진단 시작 버튼     │        │
│      └─────────────────────┘        │
│                                     │
│      입력 폼 섹션                    │
│      (직업, 연차, 소득, 자산)        │
│                                     │
└─────────────────────────────────────┘
```

#### 3.1.3 주요 컴포넌트
- `HeroSection`: 서비스 소개 및 시작 유도
- `DiagnosisForm`: 진단 입력 폼 (섹션 4 참조)
- `Footer`: 간단한 푸터 (선택)

#### 3.1.4 UX 고려사항
- **첫 화면**: 서비스의 목적을 1-2문장으로 명확히 전달
- **진행 표시**: 폼 입력 시 진행 단계 표시 (예: 1/4, 2/4)
- **입력 가이드**: 각 필드에 명확한 설명과 예시 제공
- **실시간 검증**: 입력 중 즉시 피드백 제공

### 3.2 진단 결과 페이지 (`/result`)

#### 3.2.1 목적
- 진단 결과를 명확하고 읽기 쉽게 표시
- 사용자가 자신의 상황을 깊이 생각할 수 있도록 유도

#### 3.2.2 레이아웃 구조
```
┌─────────────────────────────────────┐
│         Header (로고)                │
├─────────────────────────────────────┤
│                                     │
│      입력 요약 섹션                   │
│      (사용자가 입력한 정보 간단 표시) │
│                                     │
│      ┌─────────────────────┐        │
│      │   공통고민            │        │
│      │   (카드 형태)         │        │
│      └─────────────────────┘        │
│                                     │
│      ┌─────────────────────┐        │
│      │   현재역량            │        │
│      │   (카드 형태)         │        │
│      └─────────────────────┘        │
│                                     │
│      ┌─────────────────────┐        │
│      │   학습포인트          │        │
│      │   (카드 형태)         │        │
│      └─────────────────────┘        │
│                                     │
│      ┌─────────────────────┐        │
│      │   다시 진단하기 버튼  │        │
│      └─────────────────────┘        │
│                                     │
└─────────────────────────────────────┘
```

#### 3.2.3 주요 컴포넌트
- `ResultHeader`: 입력 정보 요약
- `CommonConcerns`: 공통고민 섹션
- `CurrentCapabilities`: 현재역량 섹션
- `LearningPoints`: 학습포인트 섹션
- `ActionButtons`: 다시 진단하기 등 액션 버튼

#### 3.2.4 UX 고려사항
- **읽기 흐름**: 위에서 아래로 자연스러운 읽기 흐름
- **시각적 구분**: 각 섹션을 카드 형태로 명확히 구분
- **타이포그래피**: 본문 텍스트는 읽기 쉬운 크기와 줄 간격
- **강조**: 중요한 문장이나 키워드는 시각적으로 강조 (볼드, 색상)
- **공간감**: 여백을 충분히 활용하여 집중도 향상
- **스크롤 애니메이션**: 섹션이 뷰포트에 들어올 때 부드러운 페이드인 효과

## 4. 컴포넌트 상세 설계

### 4.1 입력 폼 컴포넌트 (`DiagnosisForm`)

#### 4.1.1 구조
```typescript
<DiagnosisForm>
  <FormHeader />
  <JobTypeSelect />
  <CareerYearsInput />
  <IncomeInput />
  <AssetInput />
  <FormActions />
</DiagnosisForm>
```

#### 4.1.2 상태 관리
- React Hook Form으로 폼 상태 관리
- Zod 스키마로 클라이언트 사이드 검증
- 제출 시 서버 사이드 검증도 수행

#### 4.1.3 검증 규칙
- **직업**: 필수 선택
- **연차**: 0-30 범위, 숫자만 입력 가능
- **월평균소득**: 0 이상, 숫자만 입력 가능, 단위 표시 (만 원)
- **순자산**: 0 이상, 숫자만 입력 가능, 단위 표시 (만 원)

#### 4.1.4 UX 개선 사항
- **입력 필드 포커스**: 포커스 시 명확한 시각적 피드백
- **에러 메시지**: 입력 필드 바로 아래에 명확한 에러 메시지 표시
- **도움말**: 각 필드에 "?" 아이콘으로 추가 설명 제공 (호버 시 툴팁)
- **자동 포맷팅**: 숫자 입력 시 천 단위 구분 표시 (예: 1,000만 원)
- **진행 표시**: 상단에 진행 바 또는 단계 표시

### 4.2 입력 필드 컴포넌트

#### 4.2.1 `JobTypeSelect`
- **타입**: Select (드롭다운)
- **옵션**: 직장인, 프리랜서/사업자
- **디자인**: 커스텀 Select 스타일링 (기본 브라우저 스타일 대체)
- **접근성**: 키보드 네비게이션 지원, ARIA 레이블

#### 4.2.2 `CareerYearsInput`
- **타입**: Number Input
- **범위**: 0-30
- **제약**: 숫자만 입력 가능, 범위 초과 시 에러
- **UX**: 입력 시 실시간 검증, 범위 표시 (예: "0-30년")

#### 4.2.3 `IncomeInput`
- **타입**: Number Input
- **단위**: 만 원 (입력 필드 옆 또는 내부에 표시)
- **제약**: 숫자만 입력 가능, 음수 불가
- **UX**: 천 단위 구분 표시, 단위 명확히 표시

#### 4.2.4 `AssetInput`
- **타입**: Number Input
- **단위**: 만 원
- **제약**: 숫자만 입력 가능, 음수 불가
- **UX**: 천 단위 구분 표시, 단위 명확히 표시

### 4.3 결과 표시 컴포넌트

#### 4.3.1 `ResultDisplay`
- **역할**: 결과 페이지의 메인 컨테이너
- **구조**: 세 개의 섹션 컴포넌트를 순차적으로 배치
- **애니메이션**: 스크롤 시 섹션별 페이드인 효과

#### 4.3.2 `CommonConcerns`
- **디자인**: 카드 형태, 아이콘 포함 (선택)
- **타이포그래피**: 제목은 크고 명확하게, 본문은 읽기 쉬운 크기
- **레이아웃**: 최대 너비 제한 (가독성 향상)

#### 4.3.3 `CurrentCapabilities`
- **디자인**: `CommonConcerns`와 동일한 스타일, 색상으로 구분
- **내용**: 현재역량에 대한 객관적이고 구체적인 설명

#### 4.3.4 `LearningPoints`
- **디자인**: `CommonConcerns`와 동일한 스타일, 색상으로 구분
- **내용**: 학습포인트를 명확하고 실행 가능한 형태로 제시

### 4.4 UI 기본 컴포넌트

#### 4.4.1 `Button`
- **변형**: Primary, Secondary, Ghost
- **상태**: Default, Hover, Active, Disabled, Loading
- **크기**: Small, Medium, Large
- **접근성**: 키보드 포커스, ARIA 레이블

#### 4.4.2 `Input`
- **타입**: Text, Number, Email 등
- **상태**: Default, Focus, Error, Disabled
- **기능**: 에러 메시지 표시, 도움말 아이콘
- **접근성**: 라벨 연결, 에러 메시지 연결

#### 4.4.3 `Select`
- **디자인**: 커스텀 스타일링
- **기능**: 검색 가능 (선택), 키보드 네비게이션
- **접근성**: ARIA 레이블, 키보드 지원

#### 4.4.4 `Card`
- **용도**: 결과 섹션, 정보 표시
- **변형**: 기본, 강조, 그림자 효과
- **반응형**: 모바일에서 패딩 조정

## 5. 상태 관리

### 5.1 폼 상태
- **도구**: React Hook Form

### 5.2 전역 상태
- **도구**: React Context API 또는 Zustand
- **관리 항목**:

  - 로딩 상태
  - 에러 상태

### 5.3 서버 상태
- **도구**: React의 `useState`, `useEffect` 또는 React Query (선택)
- **관리 항목**:
  - API 호출 상태
  - 캐시된 결과 (선택)

## 6. API 통신

### 6.1 진단 API 호출
- **엔드포인트**: `POST /api/diagnose`
- **요청 형식**:
```typescript
{
  job_type: string;
  career_years: number;
  monthly_income: number;
  net_worth: number;
}
```
- **응답 형식**:
```typescript
{
  common_concerns: string;
  current_capabilities: string;
  learning_points: string;
}
```

### 6.2 에러 처리
- **네트워크 에러**: 사용자 친화적인 메시지 표시
- **서버 에러**: 에러 코드에 따른 적절한 메시지
- **타임아웃**: 재시도 옵션 제공

### 6.3 로딩 상태
- **제출 중**: 버튼에 로딩 스피너, 폼 비활성화
- **결과 로딩**: 결과 페이지에서 스켈레톤 UI 또는 로딩 스피너

## 7. 반응형 디자인

### 7.1 브레이크포인트
- **모바일**: 640px 이하
- **태블릿**: 641px - 1024px
- **데스크톱**: 1025px 이상

### 7.2 모바일 최적화
- **터치 친화적**: 버튼과 입력 필드 크기 최소 44px
- **가로 스크롤 방지**: 콘텐츠 너비 제한
- **입력 필드**: 모바일 키보드 타입 최적화 (숫자 키보드)
- **네비게이션**: 간단하고 명확한 구조

### 7.3 태블릿/데스크톱 최적화
- **최대 너비**: 콘텐츠 가독성을 위한 최대 너비 설정 (예: 1200px)
- **여백**: 충분한 여백으로 집중도 향상
- **레이아웃**: 2-3 컬럼 레이아웃 활용 (선택)

## 8. 접근성 (Accessibility)

### 8.1 키보드 네비게이션
- 모든 인터랙티브 요소에 키보드 접근 가능
- 포커스 표시가 명확함
- 논리적인 탭 순서

### 8.2 스크린 리더
- 의미 있는 HTML 시맨틱 태그 사용
- ARIA 레이블 및 설명 제공
- 에러 메시지가 스크린 리더에 전달됨

### 8.3 시각적 접근성
- 색상 대비: WCAG AA 기준 준수 (최소 4.5:1)
- 텍스트 크기: 최소 16px
- 포커스 표시: 명확한 시각적 피드백

## 9. 성능 최적화

### 9.1 코드 최적화
- **코드 스플리팅**: 페이지별 자동 코드 스플리팅 (Next.js 기본)
- **이미지 최적화**: Next.js Image 컴포넌트 사용
- **폰트 최적화**: 웹 폰트 최적화 (next/font)

### 9.2 로딩 최적화
- **스켈레톤 UI**: 로딩 중 콘텐츠 구조 미리 표시
- **프리로딩**: 중요한 리소스 프리로딩
- **지연 로딩**: 필요 시에만 컴포넌트 로드

### 9.3 번들 크기 최적화
- **트리 쉐이킹**: 사용하지 않는 코드 제거
- **의존성 최적화**: 필요한 라이브러리만 포함

## 11. 디자인 가이드라인

### 11.1 색상 팔레트
- **주 색상**: 신뢰감을 주는 차분한 색상 (예: 블루 계열)
- **보조 색상**: 강조를 위한 색상 (예: 그린 계열)
- **중립 색상**: 텍스트 및 배경용 그레이 스케일
- **에러 색상**: 명확한 에러 표시용 레드
- **성공 색상**: 성공 피드백용 그린

### 11.2 타이포그래피
- **제목 폰트**: 명확하고 읽기 쉬운 Sans-serif
- **본문 폰트**: 가독성 높은 Sans-serif
- **크기**: 제목 24-32px, 본문 16-18px, 작은 텍스트 14px
- **줄 간격**: 1.5-1.6 (가독성 향상)

### 11.3 간격 시스템
- **기준 단위**: 4px 또는 8px
- **일관성**: 전체 페이지에서 동일한 간격 시스템 사용

### 11.4 그림자 및 효과
- **카드 그림자**: 부드러운 그림자로 깊이감 표현
- **호버 효과**: 미묘한 색상 변화 또는 그림자 증가
- **포커스 링**: 명확한 포커스 표시
